框架选择—— Jest vs Karma

https://charith-rhettiarachchi.medium.com/why-use-jest-over-karma-for-angular-testing-b56ffa82f8

JSET建立在Jasmine之上。

Jasmine是BDD的测试开发框架。零依赖且不需要DOM

Jest是基于Jasmine构建的，拥有Jasmine所有开箱即用的功能，以及KARMA的功能。另外由于Jest是Facebook创建的，因此Facebook所有的JavaScipt库和框架，包括React。

Karma是angular团队构建的测试运行程序，在自己的浏览器环境中运行，也就是打开浏览器以运行测试。

  • Jest比Karma测试快2到3倍
  • Jest拥有快照测试
    • 首次运行测试时，可以将**.toMatchSnapshot（）**添加到组件中。
  • 更好的持续集成
    • 运行JsDOM测试，测试环境中不需要安装浏览器（由于Jest不在真正的浏览器上运行（使用jsdom）
    • 因此存在jsdom与目标浏览器不同的潜在风险*
  • 更少的配置
    • Jest的哲学之一是提供集成的“零配置”体验

关键概念

TestBed

这是 angular测试模块的基本构件块，它实际上创建了一个模拟NgModule的动态构建的测试模块，输入TestBed.configureTestingModule()的元数据也跟NgModule中的很相似。

实际上TestBed创建了一个动态测试模块，而beforeEach() 则是在每个case运行前调用的。

compileComponents()

如果要使用外部文件创建组件，例如styleUrls、templateUrl，则需要在创建组件之前从文件系统读取这些文件，这是异步操作，因此整个配置都放在异步函数中。

实际上，compileComponents是异步的从文件系统读取外部文件并编译相应的组件，如果我们使用angular cli运行测试则无需执行此操作，因为cli在测试前会编译所有的代码。

为什么需要有两个beforeEach方法？

上文提到，compileComponents 是异步函数，所以我们希望在创建用于访问组件实例的装置（fixture）前，确保组件已经编译成功。

NO_ERRORS_SCHEMA

在测试app.component时忽略所有无法识别的标签，例如page.component中的标签。如果不使用NO_ERRORS_SCHEMA ，则需要在declarations中声明所有用到的组件。

最佳实践

  • 保证单元测试在沙盒环境中运行
  • 订阅一个observable时，提供成功和失败的callback
  • 当测试组件有service依赖时，使用mock service
  • 访问DOM使用debugElement而不是nativeElement。因为debugElement提供了抽象基础运行环境。
  • 使用By.css而不是querySelector，因为querySelector只在浏览器中有效，当应用在服务上运行时会失败
  • 使用fixture.detectChanges()而不是ComponentFixtureAutoDetect，ComponentFixtureAutoDetect无法检测到同步更新
  • 不要过度使用NO_ERRORS_SCHEMA，它会忽略所有的属性和无法识别的标签

RxJS 弹珠测试

TestScheduler & RunHelpers

TestScheduler有一个名为run的回调函数，接受一个RunHelpers类型的参数，该参数提供了弹珠测试所需的所有辅助方法。

  • cold/hot ：根据给定的弹珠图创建冷/热Observable。接受一个弹珠图作为第一参数，values和errors作为附加可选参数。如果我们创建一个热的Observable，^则可将其用于指示“零帧”。
  • flush：开始一个虚拟时间。仅当在run回调之外使用帮助程序和RxJs测试工具时，或者想flush多次使用时，才需要调用。
  • ExpectObservable：用于断言Observable符合弹珠图。
  • ExpectSubscriptions：用于断言Observable具有预期的订阅。

JS 内存泄露

Posted on 2018-04-25 Edited on 2022-08-09 In keep-learning

Memory leak patterns in JavaScript

paper in 2007

IE和Mozilla Firefox：在JS中最容易内存泄露的浏览器。问题的根源是组件对象模型拿来管理dom对象，Windows的COM 组件对象模型，和Mozilla的XPCOM（跨平台组件对象模型）关于内存的分配和失而复得都用的reference-counting型垃圾收集，而reference-counting 机制并不能与JS的mark-and-sweep机制一直友好相处

JS是一个垃圾收集语言，意味着内存被释放，在对象被浏览器创建，且已经没有被引用的时候。虽然JS的垃圾回收机制没有任何毛病，但它与一些浏览器处理DOM对象的内存释放与回收不太一致。

IE和Firefox在这个问题上使用的是reference counting 的方式。 在一个reference counting 系统中，每个有引用的object 维持一个“有多少对象正在引用它”的计数，如果计数为0，这个对象将会被销毁并且内存回到堆顶。虽然这个解决方案通常非常有效，但是它有一个盲点，当它被循环引用的时候。

循环引用有什么问题呢？

循环引用是有两个互相指向对方的对象组成的，他们各给对方一个为1的计数。在一个纯粹的垃圾回收机制中，循环引用是没有问题的：如果两个被需要的对象都没有被其他任何对象引用，那么它们会被回收。在一个reference counting系统中，他们都不会被回收，因为计数永远不会达到为0。在一个混用系统中，当这两种机制一起被使用的时候，会因为系统无法区分一个循环引用而导致内存泄露，DOM对象或者JS对象都不会被销毁。

解决方案

标记-清除算法

引用计数法是将垃圾定义为没有被引用的内存，而标记-清除算法是将其定义为对象是否可以被获得。

定期从root开始向下寻找所有对象，垃圾回收记住所有的可被找到和不被找到的对象，并且将不被找到的对象回收。

从12年开始，所有现代浏览器都使用了标记-清除算法。

1 基础

QUESTION: HOW TO BUILD A NETWORK

计算机网络的主要特征是通用性。

得到网页的过程模型：

  • 域名解析(应用了UDP) => 建立TCP连接 => 浏览器发出请求，服务器返回页面 => 关闭TCP

构建计算机网络需要：

  • 确定影响网络设计的所有需求和约束。

    • 应用程序员：所需服务
    • 网络设计者： 高效
    • 网络提供者：易于执行管理

  • 连通性

    • 网络组件：节点、链路
    • 方式（从易到难）
      • 直连（一条链路，一条总线）
      • 间接连结（更多的链路，交换网络（常见类型：电路交换和分组交换），交换机（存储和转发分组））
      • 网络互联（多个网络，路由器/网关）
    • 网络互联关键
      • 为网络中每一个可到达的节点定一个地址，从而利用这个进行路由选择然后正确传送消息。

  • 有效性

    • 多路复用：多用户共享系统资源

      • 同步时分多路复用：STDM

      • 频分多路复用： FDM

      • 统计多路复用（按需而不是时间片的STDM）

        STDM：相当于传送带的固定位置占了固定的人，当接收数据的盒子经过一个人A之后，盒中数据被取出并到达下一个地方B，B装载数据，等待一个时间片的时间，再取出数据使下一个人C装载数据。其中ABC分别装载数据的时间是固定的一个时间片。给每个主机分配一段时间。

        FDM ：在传输数据的盒子中给每个主机分配一个位置，每个位置代表一个频率。

        以上两种有 时间片空闲或者位置空闲的情况，浪费资源。

        统计多路复用： 按照时间共享物理链路，传送带只有一个位置，需要的人就上前使用这个位置，并在规定的时间片内装载数据，接着数据传送给交换机，然后换下来给另外有需要的人用。

        其中，为了保证所有流都有机会传送数据，每个流在指定时间内只能传送一定大小的数据块（分组 ），超出则在这段时间都没有机会上前了。因此还需要主机（源节点）将消息化成几个分组，接收者重组还原消息。

        位置的使用权：以分组接分组为基础决定下一个发送分组的流。也相当于轮流发。

        这种方法可以避免空闲时间，使分组交换具有高效。
